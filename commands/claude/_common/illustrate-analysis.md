# illustrate-analysis — use cases + 情境矩陣分析步驟

> **載體**：[illustrate.md](../../illustrate.md) 的**分析步驟**支撐檔。use cases 分析 + 情境矩陣分析是**步驟**（可被 mode A 設計決策 / mode C 審查驗證 組合），**不是新模式**。
>
> **與 EP Scenario Matrix 的關係**：此檔定義「產出情境矩陣」的**步驟方法**（可重用）；EP 開頭的 Scenario Matrix 是此步驟的**一次產出**（該 EP 專屬 SM）— 非 兩個不同東西。

## use cases 分析步驟（從使用者日常開發，高層）

**核心原則**：use cases 從**使用者日常開發**出發（非命令內部能力）。先想「使用者在日常開發中，什麼時候有此需求」，再看命令在這些 use case 的角色。

分析流程：
1. **識別使用者角色與日常活動**（開發者做什麼：設計 / 理解 / 審查 / 溝通 / 除錯 / 維護）
2. **從日常活動抽視覺化需求**（什麼時候需要「畫出來幫思考 / 決策 / 理解 / 審查」）
3. **寫 use case**（高層：誰、什麼情境、要解什麼、命令的角色）— 非 命令內部功能

> **反模式**（避免）：從命令內部能力（如「結構 viewport」「EP 審查」）倒推 use case = 見樹不見林。use case 是使用者要解的問題，能力是手段。

## 情境矩陣分析步驟（從 use cases 抽）

**核心原則**：情境矩陣（Scenario Matrix）從 use cases 抽 — 每 use case 展開成具體情境（觸發 × 輸入 × 手段 × 預期）。

矩陣欄位：

| 欄 | 內容 |
|----|------|
| SM-N | 情境編號 |
| use case | 對應高層 UC |
| mode | 歸納的 mode（設計 / 理解 / 審查 / 溝通） |
| 觸發時機 | 什麼時候發生 |
| 輸入 | 使用者給什麼 |
| 命令手段 | 用哪些能力 / 步驟 |
| 預期產出 | 人判讀什麼 |

分析流程：
1. 每 use case 列具體情境（happy + 邊界）
2. 情境歸納 mode（同 use case 的情境可能同 mode）
3. 從情境矩陣驗證 mode 歸納（**能力跨 mode 共用 → 能力 ≠ mode**）

## 與 mode 的關係（步驟組合，非模式）

use cases + 情境矩陣分析是**步驟**，可被 mode 組合：
- **mode A 設計決策**：組合 use cases 分析（設計的 use case）+ 情境矩陣（設計情境）
- **mode C 審查驗證**：組合 use cases 分析（審查的 use case）+ 情境矩陣（審查情境）

**非模式**：use cases / 情境矩陣是分析動作，服務 mode，不是獨立產出類型。分析一個命令的 use cases + 情境矩陣（如 `/illustrate /at`）= mode（概念圖解等）**組合**這兩個步驟。
