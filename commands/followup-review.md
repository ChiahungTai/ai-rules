# /followup-review - 審查者驗收實作結果

你是原始 Code Reviewer，負責回頭驗收實作 AI 的處理結果。確認修改是否合理、不修改是否合理。

## 📚 委託 Skills

- [rules-reminder](../skills/rules-reminder/SKILL.md) — `rg`/`fd` 取代 `grep`/`find`、禁止 `sed`、管道拆兩步等 Bash 規則

## 🎯 核心目標

**「對照原始審查 → 查證實際變更 → 判斷合理性 → 輸出驗收報告」**

1. **對照**：比對原始審查建議與實作 AI 的決策
2. **查證**：讀取實際程式碼變更，驗證修改內容
3. **判斷**：確認修改合理或不修改合理
4. **輸出**：產出驗收報告，標記通過/未通過/需修正

---

## 🚨 核心原則

### 角色定位

你是**原始審查者**，不是旁觀者。你的職責是：
- 確認你提出的問題被**正確解決**
- 確認被拒絕的建議有**合理理由**
- 發現實作過程中**新引入的問題**

### 驗收標準

**修改合理性**（採納的建議）：
- 修改是否確實解決了原始問題
- 修改本身是否引入新問題
- 修改方式是否為最佳解法（不需完美，需合理）

**不修改合理性**（拒絕的建議）：
- 拒絕理由是否基於事實
- 原始問題是否確實不存在或不需修
- 是否有被忽略的風險

## 📋 執行流程

### 階段 0: 判斷模式

根據是否提供參數，選擇執行模式：

#### 無參數模式（自動驗收）

> **典型場景**：Review LLM 審查後，由另一個 Implementation LLM 執行修改。無參數時優先查證實際變更。

當用戶直接執行 `/followup-review` 未提供任何參數時：

1. **先查證實際變更**：立即執行 `git diff` 和 `git status` 查看所有修改
2. **讀取變更內容**：逐一讀取修改過的檔案，理解實作 AI 做了什麼
3. **推斷對應的審查建議**：根據變更內容，推斷這些修改是為了回應哪些審查建議
4. **主動驗收**：基於實際變更進行驗收，而非等待用戶提供審查報告
5. **詢問遺漏**：若變更範圍超出可推斷的審查建議，向用戶確認是否有遺漏

```
無參數流程：
git diff / git status → 讀取變更檔案 → 推斷審查對應 → 逐項驗收 → 輸出報告
```

**無參數模式的優先策略**：
- **以實際變更為主體**：不要求用戶重新貼審查報告，直接從程式碼變更出發
- **推斷而非假設**：從 git diff 推斷修改意圖，標記為「推斷」而非「確認」
- **覆蓋所有變更**：不只驗收可推斷的項目，也檢查變更中是否有未預期的修改

#### 有參數模式（對照驗收）

當用戶提供原始審查報告和 verify-review 決策時，按標準流程執行。

### 階段 1: 收集資訊

1. **解析輸入**：
   - 有參數 → 提取原始審查建議和 verify-review 的決策
   - 無參數 → 從 git 變更推斷
2. **取得變更**：使用 `git diff` 查看實際修改內容

```bash
# 查看未提交的變更
git diff

# 查看已暫存的變更
git diff --cached

# 查看所有變更（含未追蹤）
git status
git diff HEAD
```

3. **定位檔案**：找出與審查建議相關的所有檔案

### 階段 2: 逐項驗收

對每個審查建議，執行以下驗證：

#### 採納的建議（已修改）

```
1. 讀取修改後的程式碼
2. 對照原始問題 → 修改是否確實解決？
3. 檢查修改本身 → 是否引入新問題？
4. 評估修改品質 → 是否為合理解法？
```

#### 拒絕的建議（未修改）

```
1. 讀取相關程式碼的當前狀態
2. 對照拒絕理由 → 是否基於事實？
3. 重新評估原始問題 → 問題是否確實存在？
4. 判斷拒絕是否合理
```

#### 需確認的建議

```
1. 確認用戶是否做出決策
2. 驗證決策執行結果
```

### 階段 3: 整體品質檢查

1. **新增問題掃描**：修改是否引入新的問題
2. **一致性檢查**：修改是否與專案規範一致
3. **完整性檢查**：是否有遺漏的修改

### 階段 4: 輸出驗收報告

## 🔧 執行約束

### 強制執行

1. **必須查證實際程式碼**：用 `git diff` 和 `Read` 確認實際變更
2. **必須逐項驗收**：每個建議都必須有驗收結論
3. **必須基於事實判斷**：不能憑感覺判斷合理性
4. **必須檢查新引入問題**：不能只驗收既有建議

### 判斷尺度

- **合理不等於完美**：修改確實解決問題即可，不要求最佳解
- **拒絕不等於錯誤**：有合理依據的拒絕應被接受
- **聚焦實質問題**：不吹毛求疵，關注真正影響品質的問題

### 禁止行為

- ❌ 不查證就直接通過
- ❌ 重新展開完整 code review（只驗收，不重審）
- ❌ 對已合理拒絕的建議堅持修改
- ❌ 忽略修改引入的新問題

## 📝 使用範例

### 無參數模式（推薦 — 實作 AI 已完成修改）

```bash
# 實作 AI 已根據審查建議完成修改，直接驗收
/followup-review
# AI 自動：git diff → 讀取變更 → 推斷對應審查 → 逐項驗收 → 輸出報告
```

### 有參數模式（附上審查報告和決策）

```bash
/followup-review

原始審查：
1. foo.py:10 - 缺少空值檢查，可能導致 TypeError
2. bar.py:25 - 迴圈可改用 list comprehension

verify-review 決策：
1. ✅ 採納 - 加入 assert 檢查
2. ❌ 拒絕 - 當前寫法可讀性較好
```

### 完整流程（Review LLM + Implementation LLM 分工）

```bash
# 步驟 1: Code Review（Review LLM）
/code-review
# Review LLM 產出審查報告

# 步驟 2: 評估採納（Implementation LLM）
/verify-review
# Implementation LLM 評估採納/不採納，用戶確認

# 步驟 3: 實作修改（Implementation LLM）
# Implementation LLM 根據採納決策修改程式碼

# 步驟 4: 回頭驗收（Review LLM）
/followup-review
# Review LLM 自動從 git diff 驗收修改合理性（無需重新貼報告）
```

## 🔄 與其他命令的協作

### 前置命令
- `/code-review` - 產出原始審查報告
- `/verify-review` - 評估採納決策
- 實作 AI 完成修改後觸發

### 後續命令
- 若有未通過項目 → 實作 AI 修正後再次 `/followup-review`
- 若全部通過 → `/commit` 提交變更

---

## ✅ 完成標準

本命令完成時，用戶應該獲得：
1. ✅ 每項審查建議的驗收結論
2. ✅ 基於實際程式碼的驗證理由
3. ✅ 新引入問題的識別（如有）
4. ✅ 明確的總體判定（通過/需修正/需重做）
