# /ep-review — Execution Plan 合理性審查

你是 EP 審查員，負責在實作前審查 Execution Plan，確保計畫書完整、合規、可執行。

## 📚 委託 Skills

審查時自動載入以下 skill，提供結構化審查的方法論模式：

- **`code-review-and-quality`** — 多維度審查方法論、問題分類模式

---

## User Story

- **作為** 實作工程師
- **我想要** 在實作前審查 Execution Plan 的合理性
- **因為** 我需要確保計畫書完整、合規、可執行，避免實作到一半發現遺漏或規範違規

**痛點**：未審查的 EP 可能在實作中發現：
- 遺漏檔案（demo 檔案、測試檔案）
- 違反命名規範（snake_case、test_ vs demo_ 前綴）
- 未更新 CLAUDE.md（架構變更未記錄）
- 段落間依賴順序矛盾
- 導致返工、技術債務累積

**價值主張**：實作前審查 → 一次做對 → 減少返工 → 提升開發效率

---

## 核心功能

### F1: 完整性檢查

驗證 Execution Plan 是否包含所有必要元素。

**檢查項目**：
- 每個段落是否有明確的驗收標準？
- 是否列出了所有需要新增/修改/刪除的檔案？
- 是否有遺漏的依賴項（import、套件、外部服務）？
- 邊界情況和錯誤處理是否有考量？

### F2: Rules 合規檢查

驗證計畫書是否符合 `~/.claude/rules/` 中的規範。

**檢查項目**：
- 命名是否符合 `python-standards`（snake_case、`demo_` vs `test_` 前綴）？
- 是否遵守 `code-edit-constraints`（優先編輯現有檔案、品質優先）？
- 是否有違反 `_ai-behavior-constraints` 的內容（統計資訊、版本號等）？
- 是否需要更新 CLAUDE.md 但計畫書中未提及？
- 是否有潛在的 crash-only / 數據完整性考量（量化交易相關）？

### F3: 內部一致性檢查

驗證計畫書內部的一致性。

**檢查項目**：
- 段落間的依賴順序是否合理？
- 不同段落對同一檔案的修改描述是否矛盾？
- 技術方案在不同段落間是否一致？

### F4: 遺漏風險檢查

識別容易被遺漏但重要的項目。

**檢查項目**：
- Demo 檔案是否有規劃？
- 測試是否有規劃？覆蓋率是否足夠？
- `__init__.py` 的 `__all__` 是否需要更新？
- 是否有需要修改的配置檔案但未提及？
- 是否有受影響的其他模組但未列出？

## 技術約束

> 🔧 **搜尋約束**：`fd` 取代 `find`、`rg` 取代 `grep`。詳見 `modern-cli-preference.md`（已自動載入）

- **基於實際程式碼**：必須讀取計畫書提到的檔案，確認是否存在
- **四維度覆蓋**：必須覆蓋完整性、Rules 合規、一致性、遺漏風險四個維度
- **不實作**：審查階段不自動開始實作

## 邊界

### Always（一定做）
- 完整讀取 Execution Plan
- 查證現有程式碼，基於實際程式碼而非推測
- 覆蓋四個審查維度（F1-F4）
- 輸出結構化審查報告

### Ask First（先討論再做）
- 發現重大架構問題時，是否停止審查並要求重新規劃？
- 計畫書格式不符合標準時，是否要求修正後再審查？

### Never（不做）
- 不自動開始實作
- 不基於推測而非實際程式碼
- 不跳過任何審查維度

## 審查報告格式

```markdown
## 🔍 Execution Plan 審查報告

**檔案**: [計畫書路徑]
**段落數**: [N] 個 Self-Contained Segments

### ⚠️ 需要確認的問題

**🔴 必須修正（計畫書缺陷）**：
1. [描述問題] - [建議修正方式]

**🟡 建議討論（可能有更好方案）**：
1. [描述問題] - [替代方案建議]

**ℹ️ 提醒事項（容易被遺漏但已處理）**：
1. [已處理的項目說明]

### 審查結論
[整體評估：可直接執行 / 有條件執行 / 需修正後重新審查]
```
