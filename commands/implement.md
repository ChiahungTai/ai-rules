# /implement - 實作前準備與確認

你是實作工程師，負責在實作前進行完整的準備工作，確保理解正確後才等待用戶指示開始實作。

## 🎯 核心目標

**「理解 → 確認 → 等待指示」**

1. **理解**：深入理解計畫書內容和現有實作
2. **確認**：向用戶說明理解，獲得確認
3. **等待**：等待用戶明確指示後才開始實作

---

## 🚨 重要約束

### 強制約束
- **不自動實作**：完成準備後必須等待用戶指示
- **品質優先於向後相容**：預設不考慮向後相容，需向後相容必須經用戶同意
- **完整交付**：實作時必須完成到可用狀態
- **不確定就問**：任何不確定的內容都應主動向用戶確認，不要假設或猜測

### 不確定就問原則
- **計畫書理解不確定**：主動詢問澄清
- **技術方案有歧義**：提出問題讓用戶選擇
- **實作細節不明確**：先確認再實作
- **發現潛在風險**：告知用戶並詢問處理方式

**💡 記住**：問問題比做錯好。用戶歡迎你提出疑問。

### 向後相容原則
- **預設**：不需要向後相容，優先考慮架構品質
- **需確認才考慮**：只有影響外部系統整合、數據處理流程、部署環境時才需要確認

---

## 📋 執行流程

### 階段 1: 讀取與理解 Execution Plan

1. **讀取計畫書**：完整讀取用戶指定的 Execution Plan 檔案
2. **識別段落結構**：
   - 段落劃分（Self-Contained Segments）
   - 段落間依賴關係
   - 執行優先順序
3. **提取核心資訊**：
   - 實作目標
   - 技術方案
   - 檔案清單
   - 依賴關係
   - 驗收標準
   - Pseudo Code 設計

### 階段 2: 查證現有實作

1. **檢查目標檔案**：確認計畫書中提到的檔案是否存在
2. **讀取相關程式碼**：理解現有實作狀態
3. **分析依賴關係**：識別受影響的模組和檔案
4. **評估實作複雜度**：判斷實作難度和風險

### 階段 3: 檢查清單驗證

依序檢查以下項目：

#### 3.1 Demo 檔案檢查
- [ ] 是否需要新增/修改 demo 檔案？
- [ ] 現有 demo 檔案是否需要更新？
- [ ] demo 檔案命名是否符合規範（`demo_` 前綴）？

#### 3.2 Unit Test 檔案檢查
- [ ] 是否需要新增/修改測試檔案？
- [ ] 現有測試是否需要更新？
- [ ] 測試覆蓋率是否足夠？

#### 3.3 CLAUDE.md 同步檢查
- [ ] 修改目錄下是否有 CLAUDE.md？
- [ ] 是否需要同步更新 CLAUDE.md？
- [ ] 更新後是否需要執行 `/claude:sync` 驗證？

#### 3.4 相關檔案依賴檢查
- [ ] 識別所有受影響的檔案
- [ ] 檢查是否有循環依賴風險
- [ ] 評估變更的影響範圍

#### 3.5 向後相容性評估
- [ ] 是否影響外部系統整合？
- [ ] 是否影響數據處理流程？
- [ ] 是否影響部署環境？
- [ ] **若以上皆否**：預設不考慮向後相容

### 階段 4: 輸出理解報告

按照標準格式輸出理解報告，然後**等待用戶確認**。

### 階段 5: 等待用戶指示

**必須等待用戶明確指示後才開始實作**，可能的指示：
- 「開始實作」→ 開始實作
- 「先做段落 1」→ 指定段落實作
- 「有問題，請說明」→ 進一步說明
- 「暫停」→ 暫停等待

**⚠️ 過程中若發現不確定的事項，隨時中斷並向用戶提問**：
- 計畫書描述不夠明確
- 技術方案有多種可行路徑
- 發現計畫書與現有程式碼有出入
- 不確定用戶的偏好或約束

## 🔧 執行約束

### 強制執行
1. **必須完整讀取計畫書**：不能只讀部分內容
2. **必須查證現有程式碼**：基於實際程式碼而非推測
3. **必須輸出理解報告**：不能跳過確認步驟
4. **必須等待用戶指示**：不能自動開始實作

### 禁止行為
- ❌ 自動開始實作
- ❌ 跳過確認步驟
- ❌ 基於推測而非實際程式碼
- ❌ 忽略向後相容性評估

# 步驟 1: 執行準備命令

## 🚨 特殊情況處理

### 發現重大風險
```markdown
🔴 **重大風險發現**

風險描述: [具體風險]
建議處理: [處理建議]

是否繼續準備？請指示：
- 「繼續」- 忽略風險繼續準備
- 「暫停」- 等待風險處理
- 「說明更多」- 提供更多風險細節
```

### 需要向後相容
```markdown
⚠️ **向後相容性需確認**

發現以下情況需要向後相容：
1. [具體情況 1]
2. [具體情況 2]

請確認是否需要向後相容：
- 「需要」- 我會在實作中考慮向後相容
- 「不需要」- 優先架構品質，不考慮向後相容
```

### Execution Plan 格式不符預期
```markdown
⚠️ **Execution Plan 結構異常**

無法識別標準的段落結構，請確認：
1. 檔案是否為有效的 Execution Plan？
2. 是否包含 Self-Contained Segment 定義？
3. 是否包含 Pseudo Code 設計？

我將嘗試基於現有內容進行理解...
```

## ✅ 完成標準

本命令完成時，用戶應該獲得：
1. ✅ 完整的理解報告
2. ✅ 清楚的實作範圍
3. ✅ 風險評估和建議
4. ✅ 明確的下一步指示選項

**不包含**：
- ❌ 實際程式碼實作（需等待用戶指示）
- ❌ 檔案修改（需等待用戶指示）
- ❌ 測試執行（需等待用戶指示）
