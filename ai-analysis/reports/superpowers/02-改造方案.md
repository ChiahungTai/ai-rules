## Part 3 — 改造強化方案（雙軌 + 第二層後果）

> **單向門 / 雙向門判斷**（`deep-thinking.md`）：fork-local 改造 = 雙向門（可逆）；upstream 貢獻 = 單向門（94% rejection + 改 skill 需 eval 證據，門檻極高）。

### 路徑 A — fork-local 立即強化（雙向門，低門檻）

| 編號 | 改造 | 第一層後果 | 第二層後果 |
|------|------|-----------|-----------|
| **A1** | 補完導航層：在 root CLAUDE.md「目錄導覽」段補 `.agents/` CLAUDE.md（**Repository Architecture 不動** —— 導航文檔是後設層，非 product 架構第四層） | 導航完整，AI 進任一目錄都能定位 | fork 與 upstream 的結構敘事分歧擴大，但**這是 fork 存在的意義**（你已在做） |
| **A2** | 修 Gemini 殘留：刪 / 改 `GEMINI.md` 的 `@`-include、清 `gemini-extension.json` 或標 deprecated | fork 乾淨，Shape C 不再斷裂 | 下次 `git pull` upstream 若也修了 → 衝突；但 Gemini 已 EOL，upstream 遲早會清，衝突機率低 |
| **A3** | 加 single-source lint：`tests/` 加一個 test assert Pi `piToolMapping()` inline == `references/pi-tools.md` 內容 | 機械防護 dual-maintenance drift | 建立可貢獻 upstream 的 test 模式（路徑 B 踏腳石） |
| **A4** | 加幽靈引用 lint：掃所有 `references/<harness>-tools.md` + porting doc 引用是否存在 | CI 擋 doc-code 不同步 | 防止未來再產生幽靈引用（#2 類問題） |

### 路徑 B — upstream 貢獻（單向門，高門檻）

| 編號 | 改造 | 門檻 | 為什麼 |
|------|------|------|--------|
| **B1** | 補完 Gemini 移除（清 `GEMINI.md` + `gemini-extension.json`） | **低** | 這是「完成 upstream 自己宣告的移除」，非新功能；附 before/after 即可，不需 eval |
| **B2** | 加 doc-code 同步 lint（A3 + A4 通用版） | **中** | 純 infra test，不改 skill content，不觸 94% 紅線 |
| **B3** | 補 `.agents/` 文檔覆蓋（可選，非結構債） | **低-中** | 所有 harness manifest dotdir（1-2 檔）都沒被目錄導覽覆蓋（by design，只覆蓋 Shape B 複雜的）；若要全面覆蓋才做，需附 acceptance test |
| **B4** | 引進 ai-rules 的 A/B 雙軸 / 證據階層到 skill 內容 | **🔴 極高** | 觸「compliance changes to skills」紅線（root `CLAUDE.md:49-50`）；需大量 eval 證據；**不建議直接提** —— 走路徑 C（fork-local 實驗 → 累積證據 → 再考慮） |

### 路徑 C — 雙向借鑒（見 `03-雙向借鑒.md`）

這是**最高價值路徑**，但產出在 ai-rules 側（你的主場），不在 superpowers 側。superpowers 在這裡是「方法論來源」，不是被改的對象。

## 決策建議

見 [`README.md` 優先序表](README.md#優先序建議)。核心判斷：

1. **fork-local 先行**（P0-P1）：你的 7 個導航 CLAUDE.md + Gemini 殘留修復。這些是雙向門、低風險、完成你已啟動的方向。
2. **upstream 貢獻挑低的做**（P3）：B1（Gemini 清理）是門檻最低的 upstream 貢獻，建立紀錄。B4（引進 ai-rules 概念到 skill）門檻極高，**走 fork-local 實驗 → 累積 eval 證據 → 再考慮**，不直接提。
3. **最高價值在借鑒到 ai-rules**（P2）：superpowers 的 skill-as-TDD、progress ledger、file-handoff 是 ai-rules 缺的，且移植門檻低（在 ai-rules 主場，無 94% 紅線）。
