# sed 替換操作強烈警告

> **自動載入**: 此檔案位於 `~/.claude/rules/`，會自動載入到所有會話

> **🔴 強烈警告：禁止使用 `sed` 修改程式碼或文檔**

`sed` 是流編輯器，**完全不理解**程式碼或 Markdown 的語法結構。使用 `sed` 進行批次替換常造成災難性後果：

## ❌ sed 造成的問題

| 問題類型 | 說明 | 後果 |
|---------|------|------|
| **破壞縮排** | Python 程式碼依賴縮排，`sed` 可能破壞語意 | 程式碼邏輯錯誤 |
| **誤替換註解/字串** | `sed/g` 會替換「匹配模式」的所有出現，包括註解、字串內容 | 錯誤替換非目標內容 |
| **破壞 Markdown 結構** | `sed` 不理解標題、程式碼區塊、表格語法 | 文檔格式毀損 |
| **破壞多行結構** | `sed` 逐行處理，不理解跨行語法（括號配對、多行字串） | 程式碼結構毀損 |
| **不可預測的結果** | `sed -i` 直接修改檔案，錯誤無法還原（除非有備份） | 需要還原版本控制 |

## Edit vs Sed 選擇原理

> **核心原則**：Edit 工具能精確匹配和替換，保證只有目標區域被修改。

| 特性 | Edit 工具 | sed 命令 |
|------|-----------|----------|
| **語法理解** | 理解上下文，精確匹配 | 流編輯器，不理解文件結構 |
| **修改範圍** | 只修改目標區域 | 容易造成非預期修改 |
| **結構化文檔** | 安全處理 Markdown、程式碼 | 容易破壞結構 |
| **錯誤恢復** | 可預覽差異後執行 | `-i` 直接覆蓋，難以恢復 |

對於 Markdown 這類結構化文檔，**Edit 是更安全的選擇**。

## ✅ 正確工具選擇

| 修改目標 | 正確工具 | 說明 |
|---------|----------|------|
| **程式碼修改** | `Edit` 工具 | 需要精確替換程式碼中的特定內容 |
| **Markdown 文檔** | `Edit` 工具 | 保留標題、程式碼區塊、表格結構 |
| **變數重命名** | LSP refactoring | 需要跨檔案安全地重命名變數或函數 |
| **多行替換** | `Edit` 工具 | 替換跨多行的程式碼區塊 |
| **配置檔案** | `Edit` 工具 | 簡單 key=value 格式替換 |

## ❌ 絕對禁止使用 `sed` 的場景

- 修改任何 `.py`、`.pyx`、`.pxd` 程式碼檔案
- 修改 Markdown 文檔（`.md`，包括 CLAUDE.md）
- 修改 YAML/JSON/TOML 配置檔案
- 任何涉及程式碼語法的操作

## ✅ 唯一允許使用 `sed` 的場景

- 過濾日誌輸出（讀取後處理，不修改原始檔）
- 處理純文字資料流（無程式碼語法）
- 批次檔名重命名（使用 `rename` 更安全）

## 推薦做法：使用 Edit 工具

```markdown
## ❌ 錯誤做法：使用 sed 修改程式碼
sed -i 's/old/new/g' script.py

## ✅ 正確做法：使用 Edit 工具修改程式碼
1. 先讀取檔案理解上下文
2. 使用 Edit 精確替換目標程式碼區塊

💡 **原理說明**：sed 是流編輯器，不理解程式碼結構。Edit 工具能理解上下文並精確替換。
```
